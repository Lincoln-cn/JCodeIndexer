# Java Code Indexer 开发计划：v1.7.0 → v2.0

## 版本号规则

- 开发版：`1.7.1`、`1.7.2`、... → 所有功能开发完成 → `1.8.0` 封板发布
- 开发版：`1.8.1`、`1.8.2`、... → 所有功能开发完成 → `1.9.0` 封板发布
- 开发版：`1.9.1`、`1.9.2`、... → 所有功能开发完成 → `2.0.0` 封板发布
- 每个开发版本只提交代码、跑测试，**不打 release tag**
- 只有整数版本（1.8.0、1.9.0、2.0.0）才打 tag 触发 release

## v1.7.2 — WatchService 事件处理 + 去抖

### 开发
1. 实现事件处理逻辑：
   - `ENTRY_CREATE`：检测新文件/新目录
   - `ENTRY_MODIFY`：加入 pendingChanges 队列
   - `ENTRY_DELETE`：加入 pendingDeletes 队列
   - `OVERFLOW`：触发全量 SHA-1 比对 fallback
2. 实现去抖定时器：
   - 独立线程，500ms 间隔
   - 从队列取变更，SHA-1 验证后调用 `indexer.incrementalReindex()`
3. 修改 `Config.java`：新增 `watchDebounceMs`（默认 500）字段

### 测试
- 测试文件创建 → 检测到变化
- 测试文件修改 → 检测到变化
- 测试快速保存 → 只触发一次 reindex

# 开发规范

## 版本提交规则

- 开发版（1.7.x、1.8.x、1.9.x）：正常 commit + push，不打 tag
- 封板版（1.8.0、1.9.0、2.0.0）：更新版本号 → commit → push → 打 tag → 触发 release

## 每个开发版本的标准流程

```
1. 开发 → 编写代码
2. 单元测试 → mvn test
3. 本地打包 → mvn package -DskipTests
4. 提交代码 → git commit -m "v1.7.x: 描述"
5. 推送 → git push github/origin
```

## 每个封板版本的标准流程

```
1. 完成所有开发版本的功能
2. 更新版本号（pom.xml + McpServer + CliMain）
3. 更新 CHANGELOG.md
4. 更新 README.md / README.zh-CN.md
5. 更新 doc/api/index.md
6. 运行 mvn test → 确保全部通过
7. 运行 mvn package -DskipTests → 验证打包
8. 本地测试 Fat JAR
9. git commit → git push
10. git tag -a vX.X.X -m "版本说明"
11. git push vX.X.X → 触发 Release workflow
```
